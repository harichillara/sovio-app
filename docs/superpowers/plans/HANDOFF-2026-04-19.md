## 2. What happened in Phase 2 (the story)

Read this if you need to understand *why* the branch looks the way it does. Skip to § 3 if you just want the next task.

### Phase 2 Task 2.1 — Coordinated React 19 + SDK 53 bump (`5ffb96d`)

The plan called for a single commit bumping React 18.3 → 19.0 monorepo-wide (mobile + web + all `packages/*`) in lockstep with Expo SDK 52 → 53. This must be lockstep because Expo SDK 53 cohort-pins React 19, and having mobile on React 19 while web stays on React 18 creates a duplicate-React tree in the pnpm store → hook-call mismatches at runtime.

The agent:
1. Ran `expo install --fix` in `apps/mobile` — this bumped the SDK cohort (Expo, RN, expo-router, expo-updates, @types/react, all the expo-* packages) and critically also **downgraded** TypeScript in mobile to `~5.8.3` because that's what Expo SDK 53's peer expects. The rest of the workspace stays on `^6.0.3`.
2. Bumped `react`/`react-dom` to `19.0.0` in `apps/web/package.json`.
3. Added root `pnpm.overrides` to dedupe `react` and `react-dom`. **Tried `"$react"` sigil first, got `Cannot resolve version $react in overrides`** — pivoted to literal pins. Documented that pivot.
4. Bumped peer-dep ranges in `packages/core/package.json` and `packages/tokens/package.json` to `react: ">=19.0.0"`.
5. Left typecheck intentionally RED with 3 errors in `packages/ui` — dual `@types/react` ReactNode incompatibility (React 19's `ReactNode` includes `bigint`, which an orphan `@types/react@18.3.28` rejects).

The code review caught a real issue: the dual `@types/react` wasn't just a types drift — it was a **symptom** of an orphan `react-native@0.74.5` still living in the lockfile (dragged in by `@react-native/virtualized-lists@0.74.87`). Adding `@types/react` to overrides would band-aid the symptom; the fix was to first kill the RN orphan, which would then naturally let the right `@types/react` resolve.

### Phase 2 Task 2.2 — Fix typecheck + web build (`d7564ab`, `16ba2d5`, `9d61a27`)

This was the heavy lifting:

1. **Three pnpm overrides** added to root `package.json`, **in order** `react-native` → `@types/react` → `expo-location` (RN-first as the code review demanded). Lockfile shrank ~1500 lines confirming the prune. `pnpm -r why react-native` now returns a single version.

2. **`apps/mobile/app/_layout.tsx`** — expo-router 5 types `useSegments()` as a non-empty tuple of literal route strings, but at runtime the root path `/` yields `[]`. Widened the cast to `readonly string[]` with an explanatory comment. This makes `segments.length === 0` and `segments[0] === '(auth)'` comparisons typecheck again.

3. **`apps/mobile/tsconfig.json`** — `ignoreDeprecations: "6.0"` → `"5.0"`. TS 5.8 (SDK 53's mobile cohort pin) rejects the `"6.0"` value with TS5103. This was the **pre-existing baseline error** — changing it made Phase 2 finish *better* than baseline (typecheck EXIT=0 vs baseline EXIT=1). Self-disclosed as a scope leak in MIGRATION_LOG.

4. **Web build regression discovery.** After the agent's typecheck/test/lint fixes landed, running `pnpm build` revealed a new failure:
   ```
   Module parse failed: Unexpected token (4:7) at
   expo-modules-core@2.5.0/src/index.ts
   > export type * from './ts-declarations/global';
   ```
   `expo-modules-core@2.5.0` ships **raw TypeScript** at its entry (`"main": "src/index.ts"`), and Next 15's webpack can't parse the `export type *` syntax. The import trace was:
   ```
   apps/web/sentry.*.config.ts → @sovio/core barrel → locationService
     → expo-location → expo → expo-modules-core
   ```
   Fixed by **extending the existing `apps/web/stubs/expo-stub.js` infrastructure** (which was already stubbing `expo-secure-store`, `expo-auth-session`, `expo-linking`, `expo-notifications`, `expo-device`) to also cover `expo-location`, `expo-modules-core$`, and `expo$`. The `$` suffix uses webpack exact-match syntax to avoid intercepting deep imports. All stubbed surface is dead code on web behind `Platform.OS === 'web'` guards. Chose stubs over splitting `@sovio/core`'s barrel export because the stub pattern preserves the shared cross-platform architecture that web + mobile both depend on.

5. **Two reviews ran in parallel.** Spec review came back **PASS** with one PARTIAL finding (web-build artifact missing) — closed in `16ba2d5`. Code review came back **PASS-WITH-NOTES** with three LOW findings:
   - Typecheck artifact captured only the pnpm command echo (silent tsc success prints nothing) → re-captured with per-workspace EXIT lines in `9d61a27`.
   - `as readonly string[]` cast discards runtime nullability (`segments[0]` can be `undefined`) — deferred (comparisons are defensive).
   - Web build not wired into CI → deferred to Task 2.3.

### Phase 2 Task 2.3 — CI web-build job (`6eaf5aa`)

Added a `web-build` job to `.github/workflows/ci.yml` that runs `pnpm build` on every push/PR using the same runner/node/pnpm versions as the existing `verify` job. ## 6. How the subagent-driven workflow works (the protocol)

This migration uses a two-stage review pattern:

1. **Fresh agent implements the task.** Briefed with the task goal, scope allow-list, and known pitfalls. Produces a commit with typecheck/test/lint/bundle artifacts.
2. **Two review agents run in parallel.** One (**spec compliance**) checks that the commit matches what the plan demanded — orphans deduped, artifacts captured, scope respected. Other (**code quality**) reads the code changes for correctness, hidden risks, and rework needs.
3. **Address findings before the gate.** CRITICAL/HIGH findings block the next phase; LOW/NIT get breadcrumbed or deferred.
4. **Pause at the gate for user approval.** Don't proceed to the next phase without green light.

Precedent: Task 2.1 code review caught the orphan RN and explicitly called for RN-first dedupe order. Task 2.2 followed that discipline and landed correctly. Task 2.2 reviews both caught artifact-quality issues (web-build and typecheck) — addressed inline before advancing. **The reviews find real issues; take them seriously.**
